## Conventions de travail établies

- **Fins de ligne mixtes** : le fichier `App.jsx` et plusieurs autres contiennent un mélange de CRLF et LF selon les zones — toujours vérifier en lisant le fichier en binaire avant toute édition automatisée, ne jamais supposer un seul type de fin de ligne uniforme.
- **Nouvelles tables MariaDB** : toujours `COLLATE utf8mb4_unicode_ci`. Les tables avec FK vers `agent.cp` utilisent `ON UPDATE CASCADE ON DELETE CASCADE`.
- **Séquence de déploiement** : `npm run build` → vérifier que le hash du bundle a changé → `vercel --prod` (obligatoire, un simple build ne déploie pas) → `git add -A` → `git commit` → `git push`. Railway redéploie automatiquement le backend au push.
- **URL Vercel à utiliser** : toujours `https://f2p-pmp.vercel.app` (l'alias stable). Chaque `vercel --prod` génère AUSSI une URL unique par déploiement (ex: `f2p-xxxxxxx-canaille15-s-projects.vercel.app`) qui reste accessible indéfiniment — mais le backend (CORS, `origin: FRONTEND_URL`) n'autorise QUE l'alias stable. Un onglet resté ouvert sur une ancienne URL unique (bookmark, historique) provoque un blocage CORS silencieux côté navigateur → "Failed to fetch" à la connexion, alors que le backend fonctionne parfaitement (vécu le 10/07 : marchait sur mobile, pas sur l'onglet ordi resté sur une vieille URL de déploiement).
- **Redémarrage backend** : après modification d'un contrôleur, fermer complètement le terminal `node server.js` (pas juste Ctrl+C — un ancien process peut rester actif sur le port 3001 et servir l'ancien code) puis relancer depuis `api\api`.
- **Variables d'environnement** : `ENCRYPTION_KEY`, `JWT_SECRET`, `DB_PASSWORD` doivent être synchronisées entre `.env` local ET les variables Railway (service backend) après toute rotation — leur absence sur Railway ne fait PAS planter le serveur au démarrage mais casse silencieusement le chiffrement/l'auth en production.
- **Avant de dire "c'est fait"** : toujours vérifier par une recherche dans le fichier réel (grep/findstr) qu'un remplacement de fichier a vraiment pris, plutôt que de se fier à un message de confirmation — un vrai bug de ce soir (06/07) venait d'un fichier jamais réellement remplacé malgré plusieurs confirmations.
- **Cache navigateur** : `localStorage` clé `f2ppmp_schedule` ne se vide JAMAIS automatiquement et peut faire croire à un bug alors que c'est juste une vieille copie locale — toujours vider ce cache (ou tester en navigation privée) avant de chercher un bug d'affichage.
- **Documentation** : fichiers `FEATURES.md`, `NOTICE_ANNUAIRE.md` à la racine. Objectif à terme : une notice complète du projet, section par section, à rédiger quand le projet sera plus avancé — noter le fonctionnement détaillé (règles métier, permissions, cas limites) de chaque module au fur et à mesure.
- **Composants définis à l'intérieur d'un autre composant React** : jamais faire `const MonComposant = (props) => (...)` à l'intérieur du corps d'une fonction composant (ex: `LoginPage`) — React le recrée à chaque re-render du parent (nouvelle référence de fonction = nouveau "type"), ce qui détruit et recrée son DOM à chaque fois (perte de focus sur les champs de saisie, notamment). Toujours définir les composants au niveau racine du fichier. Piège de test associé : `preview_fill` remplit un champ en un seul coup et ne révèle pas ce genre de bug — pour tester un champ de saisie caractère par caractère, simuler une vraie frappe (valeur + `dispatchEvent('input')` + délai réel entre chaque caractère).
- **Agent de test permanent** : `0010186T` / PIN `0000`, créé par Olivier le 10/07 et à conserver tant que l'app est en développement — ne jamais le supprimer pendant les tests (utiliser un autre CP jetable pour les tests qui nécessitent la suppression, comme les tests d'admin/promotion).
- **Vérifier un correctif de warning/erreur JS dans un onglet neuf, jamais dans l'onglet déjà utilisé pour reproduire le bug** : l'outil de lecture de console du navigateur peut conserver d'anciens messages en mémoire d'une navigation à l'autre dans le même onglet — un rechargement complet, voire un redémarrage du serveur Vite, ne les efface pas forcément. Ça peut faire croire qu'un correctif n'a rien changé alors qu'il fonctionne (vécu le 11/07 sur un warning React "setState pendant le rendu"). Ouvrir un tab tout neuf avant de re-tester.
- **Numéros de ligne dans une stack trace du navigateur (Vite/HMR)** : peuvent être obsolètes après plusieurs éditions successives du fichier source dans la même session — ne pas se fier aveuglément au fichier:ligne indiqué pour localiser un bug, vérifier le contenu réel à cette ligne avant de conclure (vécu le 11/07 : une stack trace pointait vers `BarreSaisieRapide`, fausse piste — le vrai bug était ailleurs).
- **Test d'un formulaire avec touche Entrée** : un `KeyboardEvent('keydown', {key:'Enter'})` synthétique NE déclenche PAS la soumission implicite native d'un `<form>` (comportement navigateur réservé aux évenements "trusted", donc à un vrai appui clavier) — pour vérifier la logique d'un `onSubmit`, dispatcher un vrai évenement `submit` (`form.requestSubmit()`) plutôt qu'un keydown simulé.

## Modules existants (aperçu)

- **Mon Planning** (perso) : vues Mois/Semaine/Planning-liste, saisie manuelle + import bulletin de commande (PDF/photo) + import déroulé prévisionnel, note perso privée par jour, code RPP (variante de repos), compteurs (Congés/Travail/RP/RU...). La carte "Jours travaillés" du panneau compteurs est cliquable et ouvre un tableau de bord détaillé par poste (total, dernière date, détail Matin/Soir/Nuit ou Journée, répartition PRCI/PAR/Non affecté) — voir résolus du 11/07. La carte "Congés" est cliquable de la même façon et ouvre un tableau de bord dédié (droit à congés modifiable, détail mensuel, report inter-années) — voir résolus du 12/07, explication détaillée du mécanisme de report à reprendre telle quelle dans la future notice. Depuis le 15/07, Congés a en plus un cycle Demandé → Accordé/Refusé (Phase 1 d'une refonte plus large, voir résolus du 15/07) — un congé "Demandé" n'écrit jamais rien dans le planning perso, seul "Accordé" l'écrit (et écrase volontairement ce qu'il y avait). La carte "VT" (temps partiel) suit un principe voisin avec un cycle de statuts Demandé → Accordé → Pris et une synchronisation bidirectionnelle avec le planning perso — voir résolus du 14/07.
- **CPS Officiel** : planning partagé en lecture pour tous, alimenté par import PDF/photo (OCR via `ocr.space`), table `planning_cps`.
  - **Qui peut importer** : n'importe quel agent connecté, pas de restriction admin (choix assumé le 09/07 — avant ça, l'import était réservé aux admins côté backend alors que le bouton était déjà visible pour tous côté frontend, ce qui provoquait des 403 silencieusement avalés).
  - **Flux d'import** : sélection fichier → extraction OCR → écran de confirmation (récap nb agents/date/écarts) → validation explicite → sauvegarde backend → mise à jour locale seulement si la sauvegarde a réussi (plus de faux succès en cas d'échec réseau/serveur).
  - **Propagation** : `cpsSchedule` est rafraîchi toutes les 45s sur tous les appareils connectés (avant le 09/07, chargé une seule fois à la connexion — un agent déjà connecté ne voyait jamais les imports faits ailleurs sans se reconnecter).
  - **Historique & annulation** : chaque import crée un lot (`cps_import_batch` + `cps_import_detail`, snapshot avant/après par ligne). Panneau "🕓 Historique" dans l'onglet CPS Officiel, purge automatique au-delà de 90 jours (faite à chaque import, pas de tâche planifiée). Bouton "↩️ Annuler" disponible uniquement sur le tout dernier lot non déjà annulé (restaure l'état précédent ligne par ligne, ou supprime la ligne si elle n'existait pas avant).
- **Planning Prévisionnel** : import PDF "déroulé prévisionnel", extraction imparfaite connue (~257/365 jours corrects sur le seul cas testé), amélioration en attente (besoin de plusieurs autres roulements d'agents différents avant de retenter).
- **Échanges** : demandes d'échange de poste entre agents, poste/horaires auto-capturés, aucune modification auto de planning (juste un tableau d'annonces avec intérêts).
- **Fêtes légales** : suivi de prise en compte des fêtes, règles GRH00143, report N-1. Depuis le 12/07, la carte "Fêtes" du panneau compteurs ouvre un tableau de bord dédié (même principe que Congés/RP/RU), regroupé par priorité (à traiter/perdues/réglées/à venir) avec une cloche sur la carte dès qu'une fête nécessite une action — voir résolus du 12/07.
- **Annuaire** : Agents (auto-gérés) / UO (postes fixes ou liés à CPS Officiel en lecture seule, titulaire dynamique selon l'heure) / Accès rapide. Notice complète : `NOTICE_ANNUAIRE.md`.
- **Pause Figée** : suivi des pauses figées (1h30 de TC par jour), FIA (mois de prise en compte), reconstruit sur un vrai backend le 06/07 après un bug de fond (voir historique).
- **Demande de congés** : génère un PDF officiel SNCF rempli (formulaire GA_demande_autorisation_absence.pdf) via `pdf-lib` côté navigateur, + message email copiable. Suivi des demandes (tableau, statuts) pas encore construit.
- **Admin** : gestion des agents, promotion/retrait admin (effectif immédiatement depuis le 09/07, voir résolus ci-dessous), édition téléphone/email d'un agent, bascule "Réserve régionale" par agent (compteur effectif + effet de bord sur les fêtes légales, voir résolus ci-dessous).

## Ce qui a été résolu récemment (pour référence, pas pour ré-ouvrir)

- **17/07 — Bug critique corrigé : le détachement auto (Phase 3) faisait disparaître quasiment toutes les vraies demandes de congés** (signalé par Olivier : "j'ai mis 12 jours ça met 12 jours demandés mais demandée 0"). La logique de détachement écrite le 15/07 traitait TOUT jour ayant déjà un contenu dans le planning perso comme "résolu autrement" et l'excluait de la liste "Demandées" — alors que c'est le cas normal : un congé se demande presque toujours sur un jour qui a déjà une journée de travail prévue (c'est même le principe de base de la Phase 1 : la journée prévue reste affichée tant que le congé n'est pas accordé). Le bug n'avait pas été détecté en test le 15/07 car les demandes de test portaient systématiquement sur des jours vides. Corrigé en ajoutant `jourEtaitVide` (capturé au moment de la création de la demande/refus) : le détachement ne s'applique désormais que si le jour était VIDE à la demande et a été rempli depuis — jamais s'il avait déjà un contenu légitime dès le départ. Un jour réellement accordé (CA/CP) reste toujours détecté séparément via `brut`/`tousJours`, indépendamment de ce champ. Les entrées déjà en base sans ce champ (dont les 12 jours de test d'Olivier) sont automatiquement "guéries" par le correctif : `jourEtaitVide` absent = traité comme `false` = jamais détaché à tort. Vérifié en local avec une période de 4 jours chevauchant des journées de travail réelles : les 4 apparaissent bien dans "Demandées" après le correctif (0 avant). Déployé (`vercel --prod`).

- **17/07 — Race condition de synchronisation du profil corrigée** (chantier ouvert le 16/07, Olivier a validé "corrige-la d'abord" avant de reprendre Congés). Root cause : trois effets indépendants (`handleLogin`, `handleFocus`/`visibilitychange`, l'effet de chargement au login) appellent chacun `api.profil.get()` puis fusionnent la réponse dans `agentProfiles` via `setAgentProfiles`, sans jamais vérifier si un changement local a eu lieu entre le lancement de la requête et sa résolution — une réponse périmée pouvait donc écraser une modification plus récente (ex. un "Retirer" dans Congés), qui repartait ensuite vers le serveur via l'autosave, effaçant silencieusement l'action de l'agent.
  - **Correctif** : nouveau `agentProfilesRef` (miroir synchrone d'`agentProfiles` via un `useEffect`, lisible sans closure périmée depuis un callback `.then()`). Chacun des 3 sites capture `agentProfilesRef.current[agentId]` juste avant de lancer `api.profil.get()`, et compare cette référence à la valeur courante une fois la réponse reçue : si elles diffèrent (n'importe quel changement — édition locale OU fusion d'un autre fetch concurrent — a eu lieu entre-temps), la réponse est ignorée plutôt qu'appliquée. Repose sur l'immutabilité déjà en place partout dans le code (chaque édition de `agentProfiles[agentId]` crée un nouvel objet via spread, jamais de mutation en place), donc la comparaison de référence suffit sans horodatage ni `AbortController`.
  - **Vérifié en reproduisant exactement le scénario qui échouait la veille** : ajout d'une demande de congé puis "Retirer" immédiat (sans délai), rechargement à froid (`localStorage` vidé) après quelques secondes — l'entrée reste bien absente. Testé aussi en enchaînant ajout + retrait le plus vite possible (sans attendre le rendu intermédiaire) : toujours propre après rechargement à froid. L'entrée de test résiduelle du 16/07 (`congesDemandes["2026-11-20"]`) a été nettoyée avec succès via l'interface normale (elle ne revenait plus après rechargement), confirmant le correctif en conditions réelles. Déployé (`vercel --prod`).
  - **Portée** : corrige le mécanisme pour tous les champs de `agentProfiles`, pas seulement Congés/VT — bénéficie aussi à Fêtes, Pause Figée, etc.

- **16/07 — Bug critique corrigé : VT et Congés (Phases 1-4) ne s'enregistraient jamais vraiment en base** (trouvé en creusant la Phase 5, en essayant de vérifier une donnée directement depuis la base) : `src/api/client.js` — `profil.get()` et `profil.save()` construisent chacun un objet en **liste blanche explicite** de champs autorisés (`EXTRA_KEYS`) plutôt que de faire passer tout `donnees_json` tel quel.
